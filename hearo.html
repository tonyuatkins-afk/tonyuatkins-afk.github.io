HEARO

> Your machine is the protagonist.

A DOS-native music player whose north star is recognition density. Every expansion in your specific PC, every chip you remember installing, gets noticed and celebrated. Twenty-four audio devices supported, six video tiers, FPU brand recognition, a persistent Hall of Recognition that records your machine’s hardware journey across boots. Most software treats vintage PCs as generic. HEARO does not.

The unlock system

The structural heart of HEARO. Every recognized expansion unlocks specific features that are visible in the UI. The matrix is data-driven: each rule is one row in a table with a name, a description, a requirement string for the locked state, and a pure-function check against the master hw_profile_t. Adding a feature is one line.

Sixty-plus features today across six categories:

  • FPU unlocks (16): 256-bin FFT spectrum analyzer, sinc resampling for streaming, 16+ channel tracker mixing with ramping, plasma + tunnel + particle + fire + wireframe visualizers, karaoke vocal removal, 10-band parametric EQ, convolution reverb (Pentium-class), stereo widening, gamma-correct album art dithering, software quire for bit-exact mixing, adaptive-precision CORDIC, log-domain effects, FPU brand recognition for Intel, IIT 2C87, Cyrix FasMath, ULSI, and AMD parts.
  • Audio unlocks (33): RealSound PWM (PC Speaker), Tandy/PCjr PSG, Covox 8-bit DAC, Disney Sound Source FIFO, OPL2 and OPL3 MIDI, AWE64 CQM FM emulation, Sound Blaster PCM tiers (1.x mono, 2.0 auto-init, Pro/Pro 2 stereo, SB16 16-bit, SB16 ASP for QSound, full duplex on DSP 4.13+), Gravis UltraSound 32-voice hardware mixing + DRAM tier + GUS MAX line-in + CS4231 codec, Ensoniq wavetable + DAC + filters, AWE32/64 SoundFont loading + EMU8000 DSP + 32 voices + SIMM RAM, MT-32 LA mode, SoundCanvas mode (SC-55+), Yamaha XG mode, AdLib Gold 12-bit DAC and surround, MPU-401 + WaveBlaster daughterboard, Pro Audio Spectrum 16, ESS AudioDrive ESFM, Turtle Beach 56001 DSP.
  • Video unlocks (6): SVGA 1024x768x256 album-art panel, 800x600 and 640x480 SVGA tiers, 256-color cover art, high-resolution art on 1MB+ VRAM, chipset-specific accelerated blits when the SVGA chip is recognized.
  • Memory unlocks (2): Full library indexing for 100K+ tracks at 4MB+ XMS; all visualizers preloaded at 8MB+ XMS.
  • Network unlocks (5, NetISA-gated): Streaming via the ESP32 path, scrobbling, cloud library, Bluetooth output, AirPlay target.
  • FPU-less exclusives (2): Stochastic-computing particle visualizer (probability-based, no multiplications), 64-bin bipartite-table FFT spectrum. These are not consolation prizes. They run faster on a 386SX without an FPU than the FPU path runs on a 287. They are first class.

The boot screen

Every boot shows hardware detection with ENABLED highlights and unlock counts. The last line: “Your machine has unlocked NN features.” On the second and later boots, the screen leads with any new hardware detected since the previous boot. On the anniversary of the first detection of the machine fingerprint, the boot screen shows a celebratory line. Hardware detection

The detection engine is eight modules, each a small self-contained probe so a misbehaving one can be disabled with a single line change. Detection runs in dependency order at startup and produces the master hw_profile_t that every other subsystem reads.

  • CPU. FLAGS bit walk to classify 8086/186/286/386+, then CPUID when available for family/model/stepping. PIT-channel-2 timing measures actual core clock; we match against the period-nominal list (4.77, 8, 10, 12, 16, 20, 25, 33, 40, 50, 66, 75, 90, 100, 120, 133 MHz) and flag overclocked when the measurement runs more than 10% above the nearest nominal.
  • FPU. FNINIT/FNSTSW for presence, control-word width to distinguish 287-class from 387+, then brand-specific probes: IIT 2C87 mirrors a register at port 0E0Ch; Cyrix FasMath responds to a CCR3 read at port 22h/23h; integrated FPUs follow the CPU vendor.
  • Video. MDA at 3B4h vs CGA at 3D4h selects the base class. Hercules detection by 3BAh status-port toggle. EGA via INT 10h AH=12h BL=10h, VGA via INT 10h AX=1A00h, VESA via INT 10h AX=4F00h (with VBE2 prestamp for 2.0+ extras). Chipset fingerprint walks vendor-specific magic-register reads in turn: S3 (CR30 after 38h/39h unlock), Cirrus Logic (3C4h:06 = 12h unlock then 1Fh ID), Trident (3C4h:0Bh), Tseng ET4000 (3CDh write/readback), Paradise (3CEh:0Fh = 05h unlock then 0Bh ID), Oak (3DEh sentinel), Chips & Tech (3D6h sentinel).
  • Audio. The heaviest module. Always sets PC Speaker. Probes Tandy via BIOS signature at F000:FFFEh + port C0h. Walks LPT1 for Disney FIFO behavior, then Covox loopback. AdLib timer test at port 388h, with OPL3 mirror probe at base+02h/03h to distinguish OPL2 from OPL3 from AdLib Gold (OPL3 + CT1703 mixer at 38Ch). Sound Blaster family parses BLASTER environment, resets DSP, reads version: 1.x → SB 1.x, 2.x → SB 2.0, 3.x → SB Pro or Pro 2 by OPL3 presence, 4.x → SB16, plus EMU8000 probe at base+400h to identify AWE32, plus OPL3-timer-failure to identify AWE64 with CQM, plus mixer-index 83h for SB16 ASP. GUS family parses ULTRASND, resets GF1 register 4Ch at base+103h, walks DRAM, probes CS4231 codec at base+10Ch for GUS MAX. MPU-401 reset/UART-mode handshake at 330h/300h/320h/340h. Ensoniq SoundScape via OTTO chip ID. Pro Audio Spectrum 16 via mixer signature 0x50. ESS AudioDrive via SB DSP command E7h returning ESS chip ID. Turtle Beach via 56001 host port at 250h/260h/290h/320h.
  • Memory. Conventional via INT 12h. XMS via INT 2Fh AX=4300h test, then 4310h for entry point, then AH=08h call gate. EMS via INT 67h AH=42h.
  • NetISA. Card signature read at base ports 300h/320h/340h/280h, looking for NI as the first two bytes of the read-only ID register. /STUBNET command-line flag synthesises a present card.
  • Input. Mouse via INT 33h AX=0000h. Joystick via port 0x201 high-nibble pattern.
  • Orchestrator. Calls subsystems in dependency order, computes a 32-bit FNV-1a fingerprint over the stable parts of the profile, stamps the detection date.

Full probe protocols, port addresses, capability tables, and pitfalls for every device live in the soundcard reference doc.

The Hall of Recognition

Persistent text file (HEARO.HAL). One event per line: YYYY-MM-DD CATEGORY: text. A small [stats] section accumulates lifetime tallies. Plain text so the user can grep, edit, back up. Each boot, HEARO loads the file, snapshots the previous fingerprint, runs detection, compares, and appends new events.

[header]
first=2026-04-24
fingerprint=A7C3B291

[events]
2026-04-24 CPU:    Intel 80486DX2 @ 66 MHz
2026-04-24 FPU:    Integrated x87 (Intel)
2026-04-24 VIDEO:  S3 Trio64, 2MB VRAM
2026-04-24 AUDIO:  Gravis UltraSound MAX (GF1, 1MB DRAM)
2026-04-24 AUDIO:  Roland MT-32 via MPU-401
2026-04-24 AUDIO:  PC Speaker (PIT ch2)
2026-04-24 INPUT:  Mouse
2026-04-24 NETISA: not found

[stats]
boot_count=1
tracks_played=0
unique_tracks=0
hours_played_x10=0
features_unlocked=31
hardware_events=8

When you install an FPU from eBay, the next boot screen leads with “New since last boot: IIT 2C87-12 detected. 6 features unlocked.” The Hall accumulates a new FPU: IIT 2C87-12 row dated that day. A year later, on the anniversary of that detection, the boot screen reads “One year ago today, you ran HEARO on this machine for the first time.”

Anti-gamification is explicit. No achievements. No streaks. No daily nudges. Recognition, not gamification. Records, not scores.

The dormant-feature whisper

When an unlocked feature has not been used in 60+ days, HEARO may produce a single, status-bar-level whisper: “You have an MT-32. There is a Roland synth bank loaded with two unused patches.” Whispers are rate-limited: one per session, globally. They never block, never modal, never animate. They never repeat the same message twice in 90 days. The whole subsystem is ~50 lines of code; its scope is by design tiny.

Format support

Host-decoded: MIDI (SMF), WAV (PCM 8/16/24), MOD/S3M/XM/IT trackers, SID, AY, NSF, GBS, SPC.

ESP32-offloaded (NetISA path, v1.1+): MP3, FLAC, Opus, Vorbis, AAC, WMA, ALAC.

Streaming (v1.1+): Icecast/SHOUTcast, Bandcamp, SomaFM, Internet Archive, Nectarine Demoscene Radio, ModArchive.

Format honesty: HEARO does not ship an MP3 decoder that “technically runs on a 286” at 1 fps. If the host cannot do it well, the ESP32 does it; if the ESP32 is not present, the format is greyed out in the file browser.

Novel arithmetic

Six experimental techniques used in service of recognition density and graceful degradation across the CPU spectrum from 286 through Pentium:

  • Adaptive precision CORDIC. Visualizers vary CORDIC iteration count based on screen size and frame budget. A 320x200 plasma at 30 fps on a 386SX runs at 8 iterations. The same plasma at 1024x768 on a 486DX2 runs at 24 iterations.
  • Software quire. 256-bit accumulator (Posit Standard 2022 technique adapted to fixed-point). Eight 32-bit words, two’s-complement, with full carry propagation. Used when UL_EXACT_MIX is unlocked: bit-exact arbitrary-mixer pipelines.
  • Stochastic computing particles. For FPU-less systems, particle physics is approximated with biased random bits and counters. No multiplications. The FPU-less path is faster than the FPU path on slower hardware.
  • Bipartite tables. Sin/cos/log/exp at 8+ bits accuracy from two table lookups and an add. ~99% of full precision in 0.3% of the time. Total memory ~4 KB.
  • Log-domain effects. Compressor / limiter / EQ implemented in log domain (Takum 2024 technique) avoids many multiplications.
  • 8087 silicon fingerprint. The exact bit pattern of FPATAN and FYL2X differs across IIT, Cyrix, ULSI, AMD, Intel implementations. We use this to brand the FPU when CPUID is unavailable.

Suite context

HEARO is one of nine NetISA suite applications. The suite shares a config grammar (HEARO.CFG uses the same INI shape as NETISA.CFG), the Hall of Recognition file format, the boot-screen visual language, and the command-line conventions (/SAFE skips probes, /REDETECT forces a full rescan, /STUBNET provides a fake card for development).

Real-mode HEARO.EXE targets 286+ in large model. 32-bit HEAROXM.EXE targets 386+ in flat model under DOS/4GW. Both binaries ship in every release. The 32-bit binary is preferred when available; the 16-bit binary is the universal fallback. C89 strict; no C99 features; no compiler-specific extensions outside the documented Watcom __far + #pragma aux bridge in src/platform/probes.c.

The repository structure: docs/hearo-design.md (full design, 313 lines) and docs/hearo-soundcard-reference.md (probe protocols for all 24 devices, 681 lines). The hearo/ directory contains src/ (35 modules across detect, unlock, ui, math, config, platform, stub), test/ (six test executables), data/unlocks.def (manifest mirror of the unlock matrix), scripts/ (DOSBox-X screenshot toolkit), and the Watcom Makefile.

Anti-features

Not a tracker. Not a DAW. Not a podcast manager pretending to be a music player. Not gamified. Not generic. Not “minimal” in the sense of absent features; minimal in the sense of no ornamentation that distracts from the music or the machine.

FAQ

Will it run on my 286?
Yes. 80286 + 1 MB + MDA + PC Speaker is the hard floor. The text-mode UI works on every adapter from MDA through SVGA. HEARO.EXE is real-mode large-model; a 32-bit HEAROXM.EXE ships alongside for 386+ machines. No 386+ instructions leak into the 286 code path.
What audio devices are supported?
Twenty-four: PC Speaker, Tandy/PCjr PSG, Covox, Disney Sound Source, AdLib (OPL2), AdLib Gold (OPL3 + 12-bit DAC), Sound Blaster 1.x / 2.0 / Pro / Pro 2 / 16 / 16 ASP / AWE32 / AWE64, Gravis UltraSound Classic / MAX / ACE / PnP, MPU-401 plus SysEx-identified MT-32 / CM-32L / SC-55 / SC-55mkII / SC-88 / SC-88 Pro / DB50XG / SCB-55, Ensoniq SoundScape, Pro Audio Spectrum 16, ESS AudioDrive, Turtle Beach MultiSound. Probe protocols for every device live in the soundcard reference doc.
What is recognition density?
The single design principle behind HEARO. Every interaction should give the user new information about how HEARO sees their specific hardware. Boot screen, settings panel, Hall of Recognition: each one shows, in different ways, what HEARO has noticed about your machine. Most software treats vintage PCs as generic. HEARO does not.
Does HEARO play MP3 / FLAC / Opus?
Yes, but not on the host CPU. Compressed formats decode on the ESP32-S3 on the NetISA card, which delivers raw PCM back to the host over the ISA bus. Native formats (MOD, S3M, XM, IT, MIDI, WAV, SID, AY, NSF, GBS, SPC) decode on the host. Without the NetISA card, compressed formats are greyed out in the file browser. v1.0 ships native-only; v1.1 adds the ESP32 path.
How does the Hall of Recognition work?
Plain text file (HEARO.HAL). One event per line. Each boot, HEARO compares detected hardware against the previous fingerprint. New components get appended with their first-seen date. Lifetime statistics accumulate (hours played, tracks, boot count, peak features unlocked). The Hall reads like a logbook of your retro computing journey. The file is plain text so you can grep, edit, back up.
What unlocks the FFT spectrum analyzer?
Any FPU. The 256-bin FFT is one of sixteen FPU-gated features. FPU-less systems get a 64-bin bipartite-table FFT instead, which runs faster than the FPU path runs on a 287. The bipartite version is a first-class feature, not a fallback.
Is HEARO gamified?
No. The Hall of Recognition is a record, not a scoreboard. There are no achievements, streaks, or daily nudges. The dormant-feature whisper system (rate-limited to one per session) is the closest thing, and it is deliberately quiet: status-bar level, never modal, never repeating. Recognition, not gamification.
What does the NetISA card do for HEARO?
Streaming and compressed-format decode. Native HEARO runs entirely on the host CPU for tracker formats, MIDI, WAV, and chiptunes. The NetISA card adds: Internet radio (Icecast/SHOUTcast), Bandcamp, SomaFM, Internet Archive, ModArchive, plus host-side delivery of MP3/FLAC/Opus/Vorbis/AAC after decode on the ESP32. v1.0 ships without these; v1.1 turns them on.
Does it work in DOSBox-X, 86Box, PCem?
Yes. The text-mode UI uses standard BIOS calls and direct VRAM writes, both of which all three emulators support. Audio detection probes work in DOSBox-X with the standard Sound Blaster + Gravis UltraSound emulation. The screenshot toolkit at hearo/scripts/screenshot.ps1 drives DOSBox-X to the boot screen and captures it as a PNG via the existing capture.ps1 WinAPI tool.

Status
