CHIME

> Ask the network what time it is. Tell DOS.

One job, done correctly. DOS clocks drift; the CMOS battery is thirty years old; the PIT is not a frequency standard. CHIME is the small focused tool that fixes this in one keystroke when the network path is up. NetISA suite app, v1.0 foundation phase complete.

Why HTTPS Date headers, not SNTP

The protocol most people associate with network time is NTP, with its small-form sibling SNTP. Both run over UDP. The NetISA TSR INT 63h API as of v1.0 exposes TLS sessions and plaintext TCP via the session group (group 0x03), but does not yet carry UDP datagrams. CHIME v1.0 cannot speak SNTP.

What CHIME does instead: HTTPS HEAD against a small set of well-known time-anchor URLs (time.cloudflare.com, time.google.com, worldtimeapi.org), parse the response Date: header per RFC 7231 IMF-fixdate. One-second precision. The handshake is ~600 bytes; the response is ~200 bytes; the parse is character-by-character with no regex, no malloc, no scanf-style locale dependence. A worldtime JSON GET is the second-line fallback. A plain HTTP HEAD diagnostic mode lives behind /MODE=http-head for users with TLS path issues.

v1.1 adds real SNTP over UDP when NetISA exposes group 0x08, with sub-second precision via the offset/delay calculation against the server's transmit timestamp. v1.2 adds NTS-KE / NTS over UDP per RFC 8915. The HTTP-Date path stays in both versions as the diagnostic fallback; users who never enable UDP keep working without changes.

Time-set strategy

DOS exposes the system clock through three layers. The CMOS RTC at I/O ports 0x70/0x71 persists across reboots. The BIOS tick counter at memory 0040:006C runs at 18.2 Hz from PIT channel 0, and resets on reboot but reflects RTC at startup. DOS internal date/time via INT 21h AH=2Ah/2Ch get and AH=2Bh/2Dh set typically writes through to both the BIOS tick and the CMOS RTC.

"Typically" is doing a lot of work in that sentence. On most modern DOS clones (MS-DOS 5+, FreeDOS), AH=2Bh/2Dh propagate to the CMOS RTC. On some older systems (DOS 3.3, OEM 4.x), the DOS layer advances a soft clock and never touches the RTC, so a reboot loses the change. CHIME does a direct CMOS write after the INT 21h call as a backstop: stop RTC updates by setting bit 7 of register 0x0B, BCD-encode and write seconds (reg 0), minutes (2), hour (4), day (7), month (8), year-mod-100 (9), century (32, conventional), then resume updates.

Timezone is fixed-offset only (/TZ=-08:00 or in CHIME.CFG). POSIX-name DST rules are explicitly out of scope. DST is the user's problem; CHIME does not adjust. This is intentional. DST handling is one of the great rabbit holes of computing and CHIME is not the program that solves it.

The loop

You install NetISA. Your DOS clock says March 1980. You type CHIME; it asks the cloud, asks you, you say yes, the clock catches up to the present. Later, your CMOS battery is dead and the clock reset. You add CHIME /AUTO to AUTOEXEC.BAT. From then on, every boot ends with the clock correct. Periodic sync via a weekly batch job runs CHIME /AUTO /QUIET; drift never exceeds a few seconds. v1.5 ships CHIME.TSR, a 4 KB resident that hooks INT 21h time-of-day calls and silently corrects drift in the background.

Suite context

CHIME is one of nine NetISA suite apps. It inherits the suite's INI config grammar (CHIME.CFG uses the same shape as HEARO.CFG and NETISA.CFG), the command-line conventions (/SAFE, /STUBNET, /REDETECT, /VERSION, /HELP), and the deadpan tone in terminal output (one machine-readable line per fact, no boxes, no right-justify, no celebratory adjectives).

Phased roadmap

  • v1.0 Foundation (current). HTTPS HEAD + JSON GET + plain HTTP fallback. Confirmation prompt UI. INT 21h + CMOS write-through. Compilation pending Open Watcom V2.
  • v1.1 SNTP. Real SNTP over UDP when NetISA exposes the UDP group, sub-second precision, becomes the default time source.
  • v1.2 NTS. RFC 8915 secure NTP with TLS-extracted keys.
  • v1.3 TSR. CHIME.TSR resident with periodic resync, INT 21h hooks for transparent correction.
  • v1.4 GPS. Optional COM-port time source for users with serial GPS receivers (1PPS pulse plus NMEA).
  • v1.5 Multi-server. Median-of-N consensus across multiple time sources, drift trend reporting, NIST traceability claim when reachable.

Anti-features

Not a clock service. Not an NTP daemon. Not a stratum-2 server. Not a DST manager. Not a calendar app. Not a scheduled-task runner. Not a precision instrument; a one-second clock-set is acceptable for DOS and CHIME does not pretend otherwise.

The other NetISA suite apps are about doing modern things on old machines: streaming, browsing, chatting. CHIME is about doing the most basic thing on old machines: knowing what time it is. That basic thing has been broken on every retro PC for decades. CHIME is the patch.
